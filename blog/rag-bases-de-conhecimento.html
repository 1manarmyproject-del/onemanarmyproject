RAG — como dar memória real ao seu agente de IA

RAG é como dar um pendrive com os documentos certos para o modelo antes da reunião. Em vez de depender da memória de treino, o agente responde com as informações reais da sua empresa.
Fabio Xavier
Founder · OMA Project  ·  Co-criado com OMA

O problema da alucinação — e por que acontece

Você pergunta para um LLM sobre um produto específico da sua empresa. Ele responde com confiança, detalhes, preços — tudo errado. Ou pede para citar fontes e ele inventa referências bibliográficas que não existem.

Isso se chama alucinação. E não é bug — é característica do funcionamento dos LLMs.

O modelo foi treinado para sempre gerar uma resposta plausível. Quando não tem a informação específica, extrapola com padrões de linguagem. Soa certo, está errado.

A solução não é usar um modelo melhor. É não depender da memória do modelo para fatos específicos da sua empresa.

A analogia do pendrive

RAG = dar um pendrive com os documentos relevantes para o modelo antes da reunião. Ele responde com base nesses documentos, não na memória de treino.

O que é RAG

RAG (Retrieval Augmented Generation) é a técnica de buscar informação relevante numa base de conhecimento e injetar no contexto do LLM antes de gerar a resposta.

O fluxo básico:

  1. Você tem uma base de documentos — FAQs, manuais, contratos, catálogo de produtos, histórico de clientes
  2. Esses documentos são divididos em pedaços (chunks) e indexados
  3. Quando chega uma pergunta, o sistema busca os chunks mais relevantes
  4. Os chunks encontrados são injetados no prompt junto com a pergunta
  5. O LLM responde com base nessa informação real — não na memória de treino

Resultado: respostas baseadas nos seus documentos, atualizadas, com muito menos alucinação.

Busca semântica — por que vai além da palavra-chave

A parte sofisticada do RAG é a busca. Não é uma busca por palavra-chave como o Ctrl+F do Word. É busca semântica — por significado.

Quando um cliente pergunta "como cancelo minha assinatura", o sistema encontra o chunk sobre "procedimento de encerramento de contrato" — mesmo que as palavras não coincidam exatamente.

Isso é possível graças a embeddings — representações matemáticas do significado de texto. Textos com significado similar têm representações matemáticas próximas. O banco vetorial (Pinecone, pgvector) armazena e busca essas representações.

Casos de uso reais para PMEs

Atendimento ao cliente: agente que responde sobre produtos, políticas, preços — com base nos documentos reais da empresa, não na imaginação do modelo.

Assistente interno: time de vendas pergunta sobre especificações técnicas de produtos, termos de contrato, políticas de desconto — agente busca nos documentos internos e responde.

Onboarding de funcionários: novo colaborador faz perguntas sobre processos, benefícios, ferramentas — agente responde com base no manual da empresa.

RAG vs Fine-tuning — quando usar cada um

Confusão frequente: "deveria fazer fine-tuning ou RAG?"

Regra simples: use RAG quando a informação muda com frequência (catálogo de produtos, políticas, preços) ou quando você precisa de precisão factual. Use fine-tuning quando precisa ajustar o comportamento do modelo (tom, estilo, formato de resposta) e tem muitos exemplos.

Na prática, 80% dos casos de uso de PMEs são resolvidos com RAG + bom prompt — sem necessidade de fine-tuning.

"A empresa que organiza seu conhecimento interno hoje está construindo a base para os agentes de amanhã. Documentos bem estruturados são ativo de IA."
